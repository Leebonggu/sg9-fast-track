## Feedback (행동 규칙)

### 문서는 프로젝트 내 저장
모든 문서/노트는 프로젝트 `docs/` 폴더에 저장. 외부 경로(`~/.claude` 등)에 저장하지 않는다.
- 작업 요약 → `docs/YYYY-MM-DD_작업내용.md`
- 스펙/설계 → `docs/superpowers/specs/`

### v2 시트 절대 수정 금지
v2 시스템(`setup_v2.gs`)은 상용 운영 중. 기존 시트/폼 재생성·구조 변경 절대 금지. 추가/보완만 가능.
- **Why:** 실제 주민 동의 데이터가 쌓인 라이브 시스템. 재생성 시 데이터 유실.
- **How:** 새 기능은 별도 함수로 추가, 기존 함수는 hook 추가 수준으로만 수정.

### 품질 제일 원칙
확신 없는 내용은 코드를 직접 읽어서 확인 후 응답한다.
- **Why:** memory는 stale할 수 있고, 2,830세대 데이터가 흐르는 라이브 시스템에서 오답의 비용이 크다.
- **How:** memory의 파일 경로/함수명 인용 시 먼저 현재 코드 확인. 틀린 추천 시 원인 분석 후 이 파일에 기록. 추측으로 답변 금지.

### Next.js 기능 선호
배치/자동화 작업도 Apps Script 대신 Next.js(Vercel cron, API route 등) 우선 검토. 무료 범위 유지.
- **Why:** 구글 시트 배치보다 Next.js 생태계 안에서 통합 관리 선호. 유료 서비스 도입은 사전 확인 필요.

### 외부 자료·리뷰 검증 원칙
AI 리뷰·블로그·웹검색 결과는 "주장"이지 "사실"이 아님. 법령 원문·PDF·코드로 팩트체크 후 반영.
- **Why:** recon-sim 세션에서 외부 AI 리뷰를 검증 없이 옮겼다가 두 번 오답 발생.
- **How:** 검증 못 하면 "출처: X, 확인 필요" 명시. "이미 파라미터화 됨" ≠ "모델이 옳음".

### 수치 모델엔 공개 케이스로 회귀 테스트
수치 계산 모델은 공개 원문(PDF, 설명회 자료)의 입력·결과치를 회귀 테스트로 고정.
- **Why:** calculator.ts 여러 번 수정 후 중계주공5단지 PDF 케이스를 `tests/regression.ts`로 만들기 전까지는 회귀 탐지 불가능했음.
- **How:** 수치 모델 수정 후 반드시 `npm test`. 새 공개 케이스 발견 시 즉시 테스트로 추가.

## Project (프로젝트 상태)

### 통합현황 대시보드 (2026-04-26 완료)
전체 2,830세대 동의·설문 참여 현황 통합 관리 페이지.
- **아키텍처:** Materialized view — sync 시 결과를 `통합현황` 마스터 시트에 저장, 웹앱은 그것만 읽음
- **displayId 패턴:** `SurveyConfig.id`는 URL용, `displayId`는 시트 컬럼명 = UI 표시명 (그대로 일치)
  - 예: `id: 'survey-001'`, `displayId: '2026_04_기본조사_제출_완료'`
- **신속통합동의서 = 사전동의** — 동일 시스템, `동의서수거여부 === 'TRUE'`인 행만 완료
- **getMasterRows 설문 컬럼 감지:** 고정 컬럼 제외(exclusion) 방식 사용
- **코드 변경 후** 동기화 버튼 1회 클릭 필요 (마스터 시트 재작성)
- 관련 문서: `docs/2026-04-26-unified-dashboard-작업내용.md`
