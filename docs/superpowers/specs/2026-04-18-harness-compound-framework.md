# SG9 하네스-컴파운드 프레임워크

## 왜 이 프레임워크인가

SG9 프로젝트는 두 개의 시스템이 공존한다.

- **v2 라이브 시스템** (Google Forms + Sheets + Apps Script): 2,830세대의 실제 동의서 데이터가 흐르는 운영 시스템. 잘못 건드리면 데이터 손실 또는 폼 중단 사고로 이어진다.
- **웹 UI** (Next.js): 개발 중인 관리 인터페이스. 기능을 적극적으로 추가하고 수정할 수 있다.

이 두 시스템을 같은 컨텍스트에서 다루다 보면 새 Claude 세션마다 "무엇을 건드려도 되는가"를 재학습해야 하고, 실수가 발생하더라도 자동으로 막히지 않았다. 하네스-컴파운드 프레임워크는 이 문제를 해결한다.

## 하네스 엔지니어링

> 참고: https://channel.io/ko/blog/articles/what-is-harness-2611ddf1

AI를 억제하는 브레이크가 아니라, **안전하게 풀어주기 위한 제어 구조**다. 강력한 말의 힘을 유도하는 마구(harness)에서 비유된 용어.

### 3원칙

| 원칙 | 설명 | SG9 적용 |
|------|------|----------|
| **제어** | 허용 범위 밖 행동 차단 | `PreToolUse` hook이 v2 파일 수정 자동 차단 |
| **감시** | 동작 상태와 출력 추적 | CLAUDE.md 규칙 + 행동 전 자문 체크리스트 |
| **개선** | 오류 감지 → 다음 행동에 반영 | `Stop` hook → memory 교정 → 다음 세션 개선 |

### SG9 하네스 구현체

```
.claude/settings.json
├── PreToolUse hook: setup_v2.gs, setup_v2_master.gs 수정 시도 → 자동 차단 (exit 2)
└── Stop hook: 세션 종료 시 compound step 실행 프롬프트 출력

CLAUDE.md
├── Harness Rules: 절대 금지 목록 + 행동 전 자문 체크리스트
└── 품질 제일 원칙: 추측 금지, memory 교정 의무
```

## 컴파운드 엔지니어링

> 참고: https://news.hada.io/topic?id=26560

**모든 작업 단위가 이후 작업을 더 쉽게 만드는** 복리형 개발 방법론. 금융의 복리 이자처럼, 각 세션이 시스템을 학습시켜 다음 세션을 점진적으로 가속화한다.

### 4단계 루프

```
Plan → Work → Review → Compound
```

- **Plan**: 무엇을 만들 것인가, 어떤 파일을 건드릴 것인가
- **Work**: 실제 구현
- **Review**: 올바르게 동작하는가, 코드 품질은 괜찮은가
- **Compound**: **기능을 만드는 것에서 기능을 더 잘 만드는 시스템으로** — 패턴을 도구화, 학습을 memory에 기록, 반복 작업을 skill 후보로 표시

### Compound Step 체크리스트

세션 종료 전 반드시 실행 (`Stop` hook이 리마인드):

- [ ] 이번 세션에서 새로 배운 패턴/규칙이 있는가?
  - 없으면 memory에 없는 내용이면 → 기록 (feedback_, project_, user_ 타입)
- [ ] memory의 틀린 내용을 발견했는가?
  - 있으면 → 즉시 교정 (stale memory는 다음 세션에서 오답의 원인)
- [ ] 이번에 한 작업이 반복될 가능성이 있는가?
  - 있으면 → skill 후보로 표시 (MEMORY.md 비고 또는 TODO.md에 기록)
- [ ] 틀린 추천을 했는가?
  - 있으면 → 원인 분석 후 feedback memory에 기록

## 품질 제일 원칙

하네스와 컴파운드를 관통하는 핵심 원칙.

1. **확인 후 응답**: 확신 없는 내용은 코드/시트를 직접 읽어서 확인 후 응답한다. memory는 언제나 stale할 수 있다.
2. **memory 교정 의무**: memory 내용이 현재 코드와 다르면 즉시 교정한다. 틀린 memory를 그냥 두면 다음 세션에서 오답이 반복된다.
3. **오류 원인 기록**: 틀린 추천을 했다면 원인을 feedback memory에 기록한다. 자기 피드백 루프가 컴파운드의 핵심이다.
4. **추측 금지**: "아마도", "~일 것 같습니다"로 라이브 시스템에 영향을 주는 답변을 하지 않는다.

## 레이어 맵

```
새 Claude 세션 시작
        │
        ▼
┌─────────────────────────────────────────────┐
│  CLAUDE.md  (규칙 허브)                      │
│  - 하네스 규칙, 품질 원칙, 컴파운드 루프    │
│  - 프로젝트 구조, 기술 스택, 데이터 흐름    │
└──────────────────┬──────────────────────────┘
                   │
        ┌──────────┼──────────┐
        ▼          ▼          ▼
   memory/      skills/   .claude/settings.json
  (누적 학습)  (워크플로우)   (자동 가드레일)
        │          │
        ▼          ▼
  feedback_    add-survey
  project_     compound-step (후보)
  user_
```

### 각 레이어 역할

| 레이어 | 파일 | 역할 | 업데이트 시점 |
|--------|------|------|-------------|
| 규칙 허브 | `CLAUDE.md` | 행동 규칙, 프로젝트 구조 | 규칙 변경 시 |
| 자동 가드레일 | `.claude/settings.json` | v2 보호, 피드백 트리거 | 새 가드레일 필요 시 |
| 누적 학습 | `memory/*.md` | 피드백, 프로젝트 상태, 사용자 선호 | 매 세션 Compound Step |
| 워크플로우 | `skills/` | 반복 작업 자동화 | 패턴이 3회 이상 반복될 때 |
| 철학 문서 | `docs/superpowers/specs/` | 멘탈모델, 설계 근거 | 큰 결정 후 |

## 이 프레임워크를 이어받는 사람에게

이 프레임워크는 코드가 아니라 **행동 방식**을 정의한다.

- CLAUDE.md를 읽으면 무엇을 해도 되고 안 되는지 안다
- memory를 읽으면 이전 세션에서 무엇을 배웠는지 안다
- hooks가 실수를 자동으로 막아준다
- 세션이 끝날 때마다 Compound Step으로 시스템이 개선된다

**핵심**: 기능을 만들 때 동시에 "다음 사람이 더 쉽게 만들 수 있도록" 하는 것이 이 프레임워크의 목적이다.
